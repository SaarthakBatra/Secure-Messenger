# Implementation Plan - Phase 3b: Mobile Messaging Engine & Synchronization

This document outlines the design and implementation details for Phase 3b in the `modules/mobile/` package. It implements local messaging security (AES-256-GCM), live WebSocket event handling, active page conflict resolution, content-addressable hash alignment checks, Cloudflare R2 SINGLY LINKED-LIST archiving, and on-demand lazy loading.

## Detailed Workflow Examples

### 1. Cryptography Service (AES-256-GCM)
*   **Symmetric Key Identification:**
    *   The `lessonKey` is the 256-bit symmetric Conversation Key generated by the server during room initiation.
    *   This key is returned once to the creator, accepted asynchronously by the joiner, and stored securely in the client's local SQLite database under the `conversations` table as `conversation_key` (encrypted with the user's Master Storage Key, or `TranslationCacheConfig`).
    *   To perform E2EE messaging operations, the client fetches the encrypted `conversation_key` from SQLite, decrypts it in RAM using the current session's `TranslationCacheConfig` (MSK), and utilizes this plaintext `lessonKey` to derive subkeys.
*   **Normal Flow:**
    *   **Encryption:** The client derives a 32-byte messaging subkey using `SubKeyService.deriveSubKey(masterKey: lessonKeyBytes, context: SubKeyContext.messages)`. It generates a random 12-byte nonce via `Sodium.randombytesBuf(12)`. The plaintext string (e.g., `{"content":"Hello Bob","hash":"active_hash_value"}`) is encrypted using AES-256-GCM with the derived subkey and AAD `"convoId:msgId"`. The tag (16 bytes) is appended to the ciphertext. The package `(nonce || ciphertext || tag)` is Base64 encoded and returned.
    *   **Decryption:** The client decodes the Base64 ciphertext, extracts the 12-byte nonce and `(ciphertext || tag)`, and decrypts it with the messaging subkey and the exact same AAD `"convoId:msgId"`.
*   **Edge Case: Decryption Failure / Integrity Verification (E11)**
    *   If a message fails decryption (e.g., due to corrupt transport, key desync, or tampered payload), PointyCastle's `GCMBlockCipher` will throw a `CipherException`.
    *   Instead of performing a destructive database wipe, the client triggers the following recovery chain:
        1.  **Peer Re-query (P2P):** The client sends a silent WebSocket control request to the peer requesting a re-send of the specific message.
        2.  **Server Backup Fetch:** The client queries the server's `GET /conversations/:id/messages` (REST fallback) or the active page backup `GET /conversations/:id/active-page` to see if the database has a different version of the encrypted blob for that message.
        3.  **Failure Escalation:** If both sync paths fail, the client sends a failure notification control packet to the peer, throws a local decryption exception, halts rendering of the message body, and updates that message record in local SQLite with status `failed_decryption`. The UI displays an error indicator (e.g. an exclamation mark or warning badge) on the bubble.
*   **Edge Case: Memory Security**
    *   All sensitive keys in RAM (`TranslationCacheConfig`, `lessonKey`, derived messaging subkeys) are kept in `Uint8List` structures. Immediately after performing encryption/decryption, the calling method executes `keyBytes.fillRange(0, keyBytes.length, 0)` to clear the keys from memory.

### 2. Live WebSocket Service
*   **Normal Flow:**
    *   On vault unlock, the client establishes a persistent connection to `ws://localhost:3000/ws?token=$token`.
    *   When sending a message, the client serializes it to a `chat` frame and sends it over the WebSocket.
    *   When an incoming `chat` frame is received, the client:
        1.  Extracts the `messageId`, `conversationId`, `senderUserId`, and `encryptedBlob`.
        2.  Decrypts the `encryptedBlob` using `MessageCryptoService` to verify integrity.
        3.  Saves the message to SQLite with status `delivered` (or `acknowledged` if hash alignment is validated).
        4.  Instantly sends back a `receipt` frame: `{ "type": "receipt", "payload": { "messageId": "msg_id", "tickStatus": "acknowledged" } }`.
*   **Edge Case: Connection Interruption & Offline Cache**
    *   If the WebSocket connection drops (e.g., phone call interruption or network loss), the socket triggers `onDone` or `onError`. The client sets `syncStatusProvider` to `false` and starts a reconnection timer that fires every 10 seconds.
    *   During offline periods, any sent messages are stored in SQLite with status `PENDING`. When the socket reconnects, the client triggers the Active Page Sync to merge any server updates and upload local offline messages.
*   **Edge Case: Active Session Overwrite (E13)**
    *   If a new session is established on another device, the server closes the current WebSocket with code `1008` (Policy Violation, "Session overwritten").
    *   The client detects code `1008`, stops attempting auto-reconnection, wipes the local session credentials from memory, and forces the UI to redirect back to the Decoy Home screen.

### 3. Active Page Synchronization & Conflict Resolution
*   **Normal Flow:**
    *   On foreground, the client fetches the active page status (`GET /conversations/:id/active-page`).
    *   If `server_updatedAt > local_updatedAt` (Case 1): The client downloads the server's backup, decrypts the list of messages, inserts missing messages into SQLite by mapping their unique `messageId` (avoiding duplicates), and uploads a compiled merged active page to the server, updating the local and server timestamp to the current time.
    *   If `local_updatedAt > server_updatedAt` (Case 2): The client encrypts the local active messages list and uploads it via `POST /conversations/:id/active-page`, updating `server_updatedAt` to the client's timestamp.
*   **Edge Case: Simultaneous Offline Edits**
    *   Both Alice and Bob write messages while offline. Alice reconnects first and uploads her active page (`server_updatedAt` becomes $T_1$).
    *   Bob reconnects next. His local timestamp is $T_2$ (where $T_1 > T_2$ because Alice's upload is newer). Bob triggers a Case 1 sync.
    *   Bob's client downloads Alice's active page, decrypts it, and merges Alice's messages into Bob's SQLite.
    *   Bob's client then compiles the combined list of messages (Alice's merged messages + Bob's local offline messages), encrypts the list, uploads it to `/conversations/:id/active-page` with timestamp $T_3 = \text{now}$, and updates Bob's local `local_updatedAt` to $T_3$.
    *   When Alice syncs next, she downloads Bob's active page (since $T_3 > T_1$) and merges Bob's offline messages, achieving eventual consistency with zero loss.

### 4. Hash Alignment Verification
*   **Normal Flow:**
    *   Whenever a message is appended, the client queries active messages from SQLite sorted by `messageId` ascending.
    *   It extracts a list of maps containing only `{"c": decryptedContent, "id": messageId, "s": senderId, "t": timestamp}`.
    *   It serializes this list to a canonical JSON array with zero whitespace (e.g. `[{"c":"Hello","id":"m1","s":"alice","t":1716000000}]`) and computes the `SHA-256` hash of its UTF-8 bytes.
    *   When Alice sends a message, she encrypts this hash inside the message envelope. Bob receives the message, decrypts it, updates SQLite, calculates his own local active page hash, and compares it with the hash sent by Alice. If they match, Bob sends an `acknowledged` receipt.
*   **Edge Case: Parity Mismatch & P2P Sync Trigger**
    *   If Bob's computed local hash does not match Alice's transmitted hash, Bob's database has drifted (e.g., Bob missed a message while offline).
    *   Bob sends an E2EE control packet inside a `chat` frame: `{ "type": "sync_request", "lastKnownMessageId": "<oldest_active_msg_id>" }`.
    *   Alice receives the sync request, decrypts it, identifies the missing active messages, encrypts them inside a `sync_response` frame, and sends it back to Bob. Bob merges the messages and recalculates the hash to confirm alignment.

### 5. R2 History Archiver
*   **Normal Flow:**
    *   When the count of active messages in local SQLite exceeds 1000, the archiver is triggered.
    *   It queries `GET /conversations/:id/latest-chapter` to retrieve `latestChapterHash` (e.g., `hash_A`).
    *   It packages the oldest 1000 messages into a `LessonChapter` map and sets `previousChapterHash = "hash_A"`.
    *   It serializes the map, encrypts it with the `lessonKey`, and computes `new_chapter_hash = SHA256(encrypted_binary)`.
    *   It calls `POST /conversations/messages/upload-chapter-url` to get a presigned PUT URL.
    *   It PUTs the binary to the URL, then calls `POST /conversations/messages/archive-chapter` to update the tail pointer on the server to `new_chapter_hash`.
    *   On success, the client deletes the 1000 archived messages from the local active SQLite table.
*   **Edge Case: Presigned PUT Upload Failure**
    *   If the R2 upload fails (due to connection loss), the client halts the process immediately. It does not update the server pointer and does not delete local messages. It will retry the archiving routine during the next idle period.
*   **Edge Case: Pointer Update Failure**
    *   If the R2 upload succeeds but the subsequent `/archive-chapter` call fails, the server's tail pointer remains pointing to `hash_A`.
    *   On the next run, the client queries the server, gets `hash_A`, packages the same oldest 1000 messages, encrypts them, and computes the same `new_chapter_hash`. It requests the upload URL, and since R2 uploads are content-addressed and idempotent, the upload completes successfully without duplicates. The client then retries `/archive-chapter`, advancing the tail pointer and completing the sync.

### 6. Lazy Loading History
*   **Normal Flow:**
    *   The user scrolls to the top of the chat view.
    *   The client checks the oldest cached chapter's previous hash (or queries `latestChapterHash` from the server if no chapters are cached yet).
    *   It calls `POST /conversations/messages/download-chapter-url` to retrieve the presigned GET URL.
    *   It downloads the encrypted binary, decrypts it in RAM using the `lessonKey`, parses the JSON array, inserts the messages into SQLite tagged with `chapter_hash`, and prepends them to the message list in the UI.
*   **Edge Case: SQLite Cache Constraint Purging**
    *   To prevent SQLite database bloating, the database is capped to store at most the active page + 2 cached chapters.
    *   When a 3rd chapter is successfully downloaded and cached in SQLite, the lazy loading service queries `cached_chapters` table sorted by `created_at` ascending.
    *   It deletes the oldest cached chapter row from `cached_chapters` and executes `DELETE FROM messages WHERE chapter_hash = '<purged_chapter_hash>'`, releasing local storage space.

* Inspects incoming receipts (`delivered`, `acknowledged`, `read`) and updates the local database matching the `messageId`.

#### [NEW] [active_page_sync_service.dart](file:///home/saarthak.batra/Documents/Antigravity/Secure%20Messenger/modules/mobile/lib/features/messaging/services/active_page_sync_service.dart)

  - **Case 1 (Server newer):** Download `encryptedActivePage`, decrypt via `lessonKey`, write missing messages to SQLite (merging by `messageId`), combine any local offline messages, encrypt, and upload merged list via `POST /conversations/:id/active-page`.
  - **Case 2 (Client newer):** Encrypt local active messages list, POST to `/conversations/:id/active-page`.
* Automatically triggers active page upload in the background on app suspension, leaving chat, or offline recipient receipt.

#### [NEW] [hashing_alignment_service.dart](file:///home/saarthak.batra/Documents/Antigravity/Secure%20Messenger/modules/mobile/lib/features/messaging/services/hashing_alignment_service.dart)
* Builds canonical representation of active messages:
  1. Fetch active messages sorted by `messageId` ascending.
  2. Map each to: `{"c": decryptedContent, "id": messageId, "s": senderId, "t": timestamp}`.
  3. JSON serialize without whitespace.
  4. Compute `SHA-256` hash of UTF-8 representation.
* Packs `activePageHash` in outgoing message envelopes.
* Checks incoming message hashes against local hash to trigger P2P alignment routines on mismatch.

#### [NEW] [archiver_service.dart](file:///home/saarthak.batra/Documents/Antigravity/Secure%20Messenger/modules/mobile/lib/features/messaging/services/archiver_service.dart)
* Monitors SQLite capacity. If active messages count > 1000:
  1. Retrieve tail pointer `latestChapterHash` from backend (`GET /conversations/:id/latest-chapter`).
  2. Package oldest 1000 messages into `LessonChapter` object. Set `previousChapterHash = latestChapterHash`.
  3. Encrypt `LessonChapter` with `lessonKey`, compute hash `new_chapter_hash` = SHA-256 of encrypted binary.
  4. Request R2 upload URL (`POST /conversations/messages/upload-chapter-url`).
  5. Upload encrypted binary directly to R2.
  6. Finalize pointer on backend (`POST /conversations/messages/archive-chapter` with `new_chapter_hash`).
  7. Delete archived messages from SQLite.

#### [NEW] [lazy_loading_service.dart](file:///home/saarthak.batra/Documents/Antigravity/Secure%20Messenger/modules/mobile/lib/features/messaging/services/lazy_loading_service.dart)
* Triggered when scrolling to top of local history:
  1. Find oldest cached chapter's previous hash, or query `latestChapterHash` if nothing is cached yet.
  2. Query backend for presigned GET URL (`POST /conversations/messages/download-chapter-url`).
  3. Fetch encrypted binary, decrypt via `lessonKey`.
  4. Write messages to SQLite tagged with `chapter_hash = chapterHash`.
  5. Write chapter into `cached_chapters` table.
  6. Check if cached chapters count > 2; if so, delete the oldest chapter record and all its associated messages from SQLite (cache constraint).
